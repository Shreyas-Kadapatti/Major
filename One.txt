## GAP 1 ‚Äî Predictions Page Is Not Routed

**What's there:** `PredictionsPage.jsx` exists and is imported in `App.jsx`.

**What's wrong:** There is **no route** for it. `App.jsx` has no `<Route path="/predictions" ...>`. The sidebar `NAV` array in `Layout.jsx` also has no entry for `/predictions`. The page is unreachable.

**Fix needed:**
```jsx
// App.jsx ‚Äî add:
<Route path="/predictions" element={<PredictionsPage />} />

// Layout.jsx NAV array ‚Äî add:
{ to: '/predictions', label: 'Predictions', icon: Zap }
```

## GAP 4 ‚Äî No SHAP / Explainability Charts

**What's there:** `shap_enabled: True` in `_build_pipeline_config()`. The backend src has `shap_explainer.py`.

**What's missing:**
- The pipeline result dict does NOT include `shap_values` or `shap_feature_importance` ‚Äî `training_pipeline.py`'s `_build_result()` method doesn't put SHAP data in the return dict
- No `/shap` or `/explainability` endpoint in `main.py`
- No SHAP bar chart component in the frontend
- No LIME explanation table
- The Training and Evaluation pages show zero explainability content

**Fix needed:**
1. Add SHAP computation to `_build_result()` in `training_pipeline.py` (or a new endpoint that computes it on demand)
2. Add `GET /explainability` endpoint returning `{ shap_importance: {feature: value}, lime_weights: {feature: weight} }`
3. Add `SHAPBarChart` component (horizontal bar chart)
4. Wire it into the Training/Evaluation pages

## GAP 11 ‚Äî State Lost on Browser Refresh

**What's there:** All state lives in React Context (`PipelineProvider`). On startup, `prediction_service.try_load_from_disk()` runs and re-sets `state.is_trained = True` if models exist on disk.

**What's missing:**
- The frontend doesn't re-fetch results on mount if the backend says it's already trained. `refreshHealth()` is called but only fetches results if `health.pipeline_trained && !modelsData` ‚Äî however, `health` itself is fetched async and by the time it resolves, the pages may already show empty states
- On a hard refresh, `modelsData`, `hypothesisData`, and `comparisonData` are `null` even if the backend has results
- There's no "rehydrate on load" effect that checks `/pipeline-status` on app start and auto-fetches all results if `trained: true`

**Fix needed:**
Add to `PipelineProvider`'s `useEffect` on mount:
```js
const s = await api.getPipelineStatus()
setStatus(s)
if (s.trained) {
  Promise.all([api.getHypothesis(), api.getModels(), api.getComparison()])
    .then(([h, m, c]) => { setHypothesisData(h); setModelsData(m); setComparisonData(c) })
}
```

## GAP 12 ‚Äî Training Log Is Polling, Not Streaming

**What's there:** `_startPolling()` polls `/pipeline-status` every 1500ms and shows the last 20 log entries.

**What's issues:**
- Log entries are batched (only last 20 shown, backfill on each poll)
- 1.5s polling interval means up to 1.5s of visual lag on log updates
- Training can run for several minutes; polling creates ~100+ HTTP requests unnecessarily

**Original design called for:** Server-Sent Events (SSE) so log lines appear in real-time as the pipeline emits them.

**Fix needed (optional but improves UX):**
- Backend: Add `GET /pipeline-stream` that uses `StreamingResponse` with `text/event-stream` and pushes from `state.progress_log`
- Frontend: Replace polling with `EventSource` in the Training page

## GAP 13 ‚Äî Monitoring Page Uses Only One File (No Reference CSV)

**What's there:** Monitoring page uploads one "production" CSV and compares it to the original training data (which the backend reads from `state.upload_path`).

**What's missing:**
- The original Streamlit dashboard allowed uploading **both** a reference CSV and a production CSV independently
- Currently if the user uploads a new dataset and retrains, `state.upload_path` changes, breaking old drift comparisons
- No way to pin a specific reference dataset

**Fix needed:** Add a second file input on the Monitoring page for "Reference dataset (baseline)" with a fallback to the training data.

## GAP 15 ‚Äî `/models` Endpoint Missing `leakage_report` ‚Äî Backend Bug

**What's there:** The `get_models()` handler tries to call `leakage_report.to_dataframe()` and read `.blocked`, `.warnings`, `.clean` attributes.

**What's wrong:** The `training_pipeline.py`'s `_build_result()` method does **not include `leakage_report`** in the returned dict. The pipeline stores `self.leakage_report` internally but it's never put into the result dict. So `r.get("leakage_report")` is always `None`, meaning leakage data in the Training page is always blank.

**Fix needed:** In `training_pipeline.py`, add `"leakage_report": self.leakage_report` to the `_build_result()` return dict. Also verify that `leakage_report` has the `.blocked`, `.warnings`, `.clean`, `.to_dataframe()` interface.

---

## GAP 16 ‚Äî `@tanstack/react-query` and `zustand` Are Not Used

**What's there:** `axios` is imported in `package.json` but never used (all requests use `fetch`). `@tanstack/react-query` and `zustand` are not in `package.json` at all.

**What's impact:**
- No request caching, no automatic refetch-on-focus, no loading/error state management per query
- Manual state in `PipelineProvider` is verbose and duplicates React Query's job
- Not a blocker but makes caching, background refetching, and stale data harder to manage

**Fix needed (optional):** Add `@tanstack/react-query` as a dependency and migrate `api.js` calls to `useQuery`/`useMutation` hooks for the results endpoints.
